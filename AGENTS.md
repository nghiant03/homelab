# AGENTS.md

## Repository purpose

This is a Kubernetes GitOps homelab repository. It contains plain YAML manifests grouped into Kustomize roots under `apps/` and `platforms/`; there is no application source code, package manager metadata, Makefile, CI workflow, or repo-local script layer.

Flux is configured to reconcile this repository from Soft Serve over SSH. The generated Flux sync manifest points at `main` and sets the Flux `Kustomization` path to `./platforms`:

- `platforms/flux-system/gotk-sync.yaml` defines the `GitRepository` and Flux `Kustomization`.
- No `platforms/kustomization.yaml` was observed, so verify/repair this before assuming Flux can render the configured path.
- Manifests under `apps/` are currently not wired into Flux by any observed top-level Kustomization.

## Layout and control flow

- `platforms/flux-system/`
  - Flux bootstrap manifests.
  - `gotk-components.yaml` and `gotk-sync.yaml` are generated by Flux and marked `DO NOT EDIT`.
  - `kustomization.yaml` includes only the generated Flux files.
- `platforms/tailscale-operator/`
  - Installs the Tailscale Kubernetes operator, its CRDs, RBAC, service accounts, and the `tailscale` `IngressClass`.
  - `deployment.yaml` runs `tailscale/k8s-operator:stable` in namespace `tailscale` and reads OAuth credentials from the `operator-oauth` Secret.
  - `crd/kustomization.yaml` lists the Tailscale CRDs separately and is included from the operator root.
- `platforms/soft-serve/`
  - Runs `charmcli/soft-serve:v0.11.6` in namespace `git` with a `20Gi` PVC.
  - Exposes an internal ClusterIP service for SSH/HTTP/stats/git ports and a Tailscale LoadBalancer service named `soft-serve-tailnet` for SSH.
  - The Flux source URL in `platforms/flux-system/gotk-sync.yaml` points to `ssh://flux@soft-serve.tail36f6a3.ts.net:23231/homelab.git`, so Soft Serve is part of the GitOps loop.
- `apps/whoami/`
  - A simple Traefik whoami deployment, service, namespace, and `whoami.home.arpa` ingress.
- `apps/homepage/`
  - Homepage deployment, service account, cluster RBAC, config ConfigMap, service, namespace, and two ingress entries.
  - `homepage-traefik` serves `homepage.home.arpa`; `homepage-tailscale` uses `ingressClassName: tailscale` with TLS host `homepage`.

Each immediate app/platform directory is a standalone Kustomize root with its own `kustomization.yaml`. Add new manifests to the relevant directory's `resources` list or they will not be rendered by Kustomize.

## Commands

No committed Makefile, scripts, package metadata, CI workflows, or documented test commands were observed.

```sh
kustomize build apps/whoami
kustomize build apps/homepage
kustomize build platforms/soft-serve
```

- `platforms/flux-system/gotk-sync.yaml`

## Secrets and SOPS

SOPS is configured in `.sops.yaml`:

- Every `*.yaml` / `*.yml` path matches `path_regex: .*\.ya?ml$`.
- Only `data` and `stringData` fields are encrypted (`encrypted_regex: "^(data|stringData)"`).
- The configured recipient is an age key.

Observed encrypted Secrets:

- `platforms/soft-serve/secret-admin-key.yaml` contains `stringData.admin_keys` for Soft Serve initial admin keys.
- `platforms/tailscale-operator/secret-operator-key.yaml` contains `stringData.client_id` and `stringData.client_secret` for the Tailscale operator OAuth Secret.

Do not replace encrypted `ENC[...]` values with plaintext in committed manifests. When creating or changing a Secret, keep the Secret manifest in place and encrypt `data`/`stringData` through SOPS before commit.

## Manifest conventions observed

- Resource names usually match the directory/component name: `whoami`, `homepage`, `soft-serve`.
- Workload selectors commonly use `app.kubernetes.io/name: <name>` for app workloads; the Tailscale operator manifests use `app: operator` because they appear to follow upstream-generated structure.
- Most components create and reference their own namespace in the same Kustomize root.
- Container images are pinned to explicit tags for `whoami`, Homepage, and Soft Serve; Tailscale operator/proxy currently use `stable`.
- Services generally target named container ports when the deployment names ports; the whoami manifests use numeric port `80` without a named port.
- YAML formatting is not fully normalized across the repository: some manifests use 2-space indentation, while Tailscale operator and SOPS-encrypted Secret files use 4-space indentation in places. Preserve local style when editing existing files unless intentionally reformatting a whole manifest.

## Ingress and exposure patterns

- `.home.arpa` hostnames are used for local ingress hosts (`whoami.home.arpa`, `homepage.home.arpa`).
- Tailscale exposure uses either:
  - `ingressClassName: tailscale` for an Ingress, as in `apps/homepage/ingress.yaml`, or
  - `type: LoadBalancer` plus `loadBalancerClass: tailscale`, as in `platforms/soft-serve/service.yaml`.
- The Tailscale operator defines the `tailscale` `IngressClass`; do not use Tailscale ingress/load balancers without ensuring `platforms/tailscale-operator/` remains applied first.

## Gotchas for future changes

- `apps/` is not referenced by the observed Flux sync path. Adding an app under `apps/` and its local `kustomization.yaml` is not enough to deploy it unless a Flux/Kustomize path is added elsewhere.
- `platforms/flux-system/gotk-components.yaml` is large generated YAML; avoid broad formatting or search/replace edits there.
- `platforms/flux-system/gotk-sync.yaml` is generated and also contains the Git source of truth. Editing it manually can break reconciliation or be overwritten by Flux bootstrap output.
- The repo has no top-level aggregator `kustomization.yaml`; validate each root directory independently.
- SOPS configuration applies to all YAML files, but only `data` and `stringData` fields are encrypted. Non-secret fields in Secret manifests remain readable and should not contain sensitive material.
- Homepage needs its service account and ClusterRole/ClusterRoleBinding for Kubernetes widgets; removing `serviceAccountName: homepage` or `apps/homepage/rbac.yaml` will likely break cluster visibility in Homepage.
- Soft Serve depends on both its PVC and `soft-serve-admin-key` Secret. Its data path is set by `SOFT_SERVE_DATA_PATH=/soft-serve` and mounted from the PVC.

## Adding a new component

When adding a new app or platform component, follow the observed structure:

1. Create a dedicated directory under `apps/` or `platforms/`.
2. Include a `namespace.yaml` if it runs in its own namespace.
3. Create a local `kustomization.yaml` listing every manifest file in that directory.
4. Use matching labels/selectors between Deployment and Service.
5. Add any Secret values as SOPS-encrypted `data` or `stringData`.
6. If the component must be reconciled by Flux, make sure it is reachable from the Flux-applied path, not just present in the repository.
